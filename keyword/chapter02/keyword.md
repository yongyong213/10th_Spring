- **RESTful API란?**
    
    REST 원리를 따르는 API
    
    **REST의 특징**
    
    - 서버-클라이언트 구조
    - Stateless(무상태)
        - 서버는 클라이언트 상태를 저장하지 않고, 요청에 다 담겨 있어야 한다.
        - 각각의 요청이 독립적이다.
    - 캐시처리가능
        - HTTP의 기본적인 캐싱 매커니즘을 사용할 수 있다.
        - 클라이언트나 프록시에서 데이터를 캐싱하여 응답 속도를 높일 수 있다.
    - 계층화
    - 인터페이스 일관성
- **멱등성이란?**
    
    연산을 여러 번 적용하더라도 결과가 달라지지 않는 성질 **(서버의 상태)**
    
    |  | **멱등성** |
    | --- | --- |
    | **GET** | O |
    | **POST** | X |
    | **PUT** | O |
    | **PATCH** | X |
    | **DELETE** | O |
    - GET과 DELETE는 단순 조회, 삭제이기 때문에 멱등성 만족
    - PUT도 대상 리소스를 추가하거나 있다면 덮어 씌우기 때문에 멱등성 만족
    - POST의 경우 항상 데이터를 추가하게 되므로 멱등성을 만족하지 않는다.
        - 같은 정보를 계속 POST하면 계속 추가됨.
    - 수정할 리소스의 일부만을 PATCH로 담아보내면 멱등성이 보장되지만, PATCH의 동작이 ‘증가’와 같은 경우에는 멱등성이 보장되지 않는다.
- **HTTP 메서드 종류**
    
    **주요 메서드**
    
    - GET
        - 멱등성O
    - POST
        - 바디를 통해 서버로 요청 데이터 전달
        - 멱등성X
    - PUT
        - 리소스를 대체(수정)하는 메서드
        - 리소스가 있다면 덮어쓰고, 없으면 새로 생성
        - 멱등성O
    - PATCH
        - 리소스의 일부분을 변경
        - 멱등성X
        - PATCH를 지원하지 않는 서버에서는 POST로 대체할 수 있음
    - DELETE
        - 리소스 제거
    
    **기타 메서드**
    
    - HEAD
        - GET과 동일하지만, 서버에서 바디를 리턴하지 않음
        - 오직 찾기만 (검사)
    - TRACE
        - HEAD와 마찬가지로 일종의 검사용 메서드
        - 서버에 도달 했을 때의 최종 패킷의 요청 패킷 내용을 응답 받을 수 있음
        - 200 응답의 내용(body)로 수신한 메세지를 송신자에거 반송한다.
    - OPTION
        - 예비 요청에 사용되는 메소드
        - 본 요청을 하기 전에 안전한지 미리 검사
        - CORS 정책을 검사하기 위한 요청
    - CONNECT
        - 터널링: 클라이언트와 서버 사이에 데이터를 그대로 통과시키는 통로 만들기
        - 암호화된 통신을 할 때 사용
